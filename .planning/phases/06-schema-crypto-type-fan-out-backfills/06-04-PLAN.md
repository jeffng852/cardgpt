<objective>
Retire the write-only dead `RewardCap` schema at the data layer (TECH-02). The interface + field were removed in 06-01; this plan removes the 3 populated data usages (citi-cash-back, sc-smart, sc-simply-cash) plus the template/example/README references, and PROVES the retirement is output-neutral with a byte-identical before/after capture of the recommender's ranking for those 3 cards.

Why this is safe / what "migrate to the live per-rule cap model" means here: the shipping cap model is per-rule `monthlySpendingCap` + `fallbackRate` (already fully wired and read by the engine). The 3 `rewardCap` objects contain only `redemptionThreshold` (a redemption floor, no per-rule equivalent, read by nothing) and one `monthlyLimit` on sc-smart (a reward ceiling — NOT a spend cap; also read by nothing). The engine references neither `rewardCap` nor the `RewardCap` type (grep-proven), so removing the data cannot change any ranking. The live per-rule model is declared canonical; the dead reward-side/redemption values are dropped (not ported), because they are unrepresentable in the spend-cap model and were never honored. The byte-identical diff is the empirical proof; the grep is the deterministic backstop.

Purpose: Eliminate a data-implied contract nothing honors, provably without regressing recommendations.
Output: rewardCap-free corpus/docs + a reproducible ranking-capture script.
</objective>
