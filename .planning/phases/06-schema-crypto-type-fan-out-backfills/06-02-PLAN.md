<objective>
Fan out the widened `RewardUnit` (from 06-01) across the recommendation engine, data-layer validation, and AI extraction so a `crypto` reward "crashes nothing" (CRY-02). The single highest-risk site is `groupByRewardUnit` in `recommendCards.ts`: it builds `Record<'cash'|'miles'|'points', ...>` and does `grouped[rec.calculation.rewardUnit].push(rec)` â€” a crypto reward makes `grouped['crypto']` undefined and `.push` throws. This plan fixes that undefined-key path and every other closed-set site in the engine/data/AI layers.

It also lands the CRY-01 validation rule: `validateCard` must reject a card with no (or invalid) `cardType`, satisfying SC1's "a card missing a type fails validation." This lives here because it sits beside `validateRewardRule` in the same file.

Purpose: Guarantee the recommender and its validators tolerate crypto data before any crypto card exists.
Output: Crash-proof engine + validators that accept crypto units and require cardType.
</objective>
